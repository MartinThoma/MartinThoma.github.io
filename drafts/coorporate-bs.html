I give 110%

And similar statements like "I want to be 200% certain". Yes, I get it, they use a quantitative statement to express the qualitative statement like "much" or "very". Still, this is bullshit.

While being more than 100% convinced of something doesn't make any sense, giving 110% actually can have a meaning that does: If 100% is the amount that is sustainable, 110% would be something that can be given only a short time. So this could be a way to express that the current work-load will lead to burn-out.

There are no bad ideas

The statement is often made in a brainstorming atmostphere. It is used to express that one has the ability to freely think without having to worry about all of the potential hurdles.

It does make sense to keep the potential problems aside for a moment, if they are possible to solve at all.

It is bullshit to say that there are no bad ideas. There are obvious ones - genocide I'd say is one. Good luck with arguing that this is not a bad idea. Now you might think "yeaaaaah ... but no problem related idea". This problem related addendum, however, can also mean that all unrelated ideas are bad. Then you have to define the border what is problem-related and what isn't. Anyway, you might say that we are all adults and we know what is problem-related and what not. I agree. With the same argument, I would say that there are usually some unsolvable known problems and some known hard problems. Not considering them in the brainstorming process at all always seemed weird to me.

Everybody in the team should be able to pick up any story

There are a couple of ideas which I'd say are usually not combinable:

  • Everybody in the team should know what to do for a given story.
  • Everybody in the team should know how to do the "what" of a given story.
  • Everybody in the team should be able to pick up any story in the sprint

OOP

Don't get me wrong. OOP makes perfect sense in some cases. But I've seen way to many cases where people over-engineer stuff because they want to use OOP so desperately. Jack Diederich makes this point in his talk Stop Writing Classes. Aparently he came to a Python project with 660 lines of code, 20 modules and 20 classes which he could reduce to 5 lines of code and a single function which only uses core library function calls. I have seen code bases where I have the strong suspicion that the same is the case. Usually Java (or at least written by people with a Java background) who learned that it is "good style" to abstract away as much as possible if "you might need to extend it in future". YAGNI is the princple I and many Python programmer like to follow. Or, to say it with the Zen of Python:

Simple is better than complex. [...] Flat is better than nested. [...] Readability counts.

An awesome example Jack Diederich points to is Conways Game of Life. It can be implemented efficiently with one function and a set / a dictionary.
